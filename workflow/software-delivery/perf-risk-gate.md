# Performance Risk Gate（PR 階段效能風險檢查流程）

`workflow/software-delivery/perf-risk-gate.md` 是 software-delivery 主 flow 的一個 sub-step，當 PR 觸動效能敏感路徑（特別是 AI 生成程式碼大量進入時）必須執行。本文件只放執行步驟；抽象原則在 [`intelligence/engineering/ai-augmented-delivery/generation-validation-rate-parity.md`](../../intelligence/engineering/ai-augmented-delivery/generation-validation-rate-parity.md)；具體 detection rule 在 [`validation/scenarios/software-delivery/ai-codegen-perf-risk-checklist.yaml`](../../validation/scenarios/software-delivery/ai-codegen-perf-risk-checklist.yaml)。

## 何時進入此 Gate

PR 滿足**任一**條件即進入：

- Diff 觸動熱路徑：API handler、DB 查詢層、外部 HTTP 呼叫、cache / session 累積結構
- PR 有 AI 標記：`Co-Authored-By: <model>`、"generated by"、"AI-assisted"
- PR 描述只提到 functional，未提到 perf / load / latency / memory
- Reviewer 主觀判斷該 PR 量大或結構複雜，且無 perf 證據附件

Opt-out：在 PR / commit body 加 `[skip-perf-bench]` 並附正當理由（純文件、純 UI 樣式、純測試、純機械 refactor）。

## 執行步驟

### Step 1：靜態 perf anti-pattern 掃描（PR 開啟時）

對應 [`ai-codegen-perf-risk-checklist.yaml`](../../validation/scenarios/software-delivery/ai-codegen-perf-risk-checklist.yaml) 中四項 check：

1. `no_query_in_loop` — 迴圈內含 DB 呼叫（N+1）
2. `bounded_collections` — Long-lived collection 無 eviction
3. `external_call_has_timeout` — HTTP / RPC 無 timeout
4. `parameterized_sql` — SQL 字串拼接

任何一項失敗：PR 標記 `perf-gate-blocked`，需修復或 opt-out 才能繼續。

### Step 2：熱路徑 micro-benchmark（PR ready-for-review 時）

若 Step 1 通過但 diff 仍觸動熱路徑：

1. 識別熱路徑 entry（typically API handler 或 DB 操作）
2. 對該 entry 寫一個 micro-benchmark（Go: `BenchmarkXxx`、Python: pytest-benchmark、Node: bench/v8 hint）
3. 跑 baseline（merge target）+ candidate（PR HEAD），attach 兩組數據
4. 若 candidate 比 baseline 慢 > 10%（threshold 可調），標記 `perf-regression-flag`，reviewer 必須 explicitly accept

### Step 3：Reviewer perf checklist（review 階段）

Reviewer 不僅看「功能對不對」，必須在 PR comment 中明確回答四個問題：

1. **Load 下會怎樣？** 1k QPS / 10k 筆資料 / 100MB 流量下這段 code 的行為？
2. **邊界 case？** 空集合、單筆、極大集合、null、超時、下游錯誤？
3. **失敗 cascade？** 這段呼叫的下游若慢回應或失敗，對整個 service 的影響？
4. **Observability？** 出問題時，operator 能從現有 metrics / tracing / log 抓到根因嗎？

四題全部回答後才能 approve。

### Step 4：Pre-deploy observability gate（merge 後、deploy 前）

該 PR 觸及的路徑必須有：

- **Metrics**：request rate、error rate、latency p50/p95/p99 已 instrumented
- **Tracing**：對應 endpoint 有 distributed tracing
- **SLO alerting**：對應 SLO 已配 alert（latency 超標 / error rate 超標）

若無，要先補上才能 deploy。對應 [`analysis/production/`](../../analysis/production/README.md) 的 production 分析方法。

### Step 5：Canary / progressive rollout（deploy 階段）

避免 full deploy 後才發現問題：

- 1% → 10% → 50% → 100% 漸進
- 每個階段觀察 SLO 是否違反
- 違反即 auto-rollback

## 失敗處理

| 階段 | 失敗訊號 | 處理 |
|---|---|---|
| Step 1 | 4 項 anti-pattern 任一被命中 | 修復原始碼或加 opt-out 並說明 |
| Step 2 | Benchmark 顯示 > 10% regression | Reviewer explicit accept + 標註原因 OR 修復 |
| Step 3 | Reviewer 4 題未答完 | 不能 approve |
| Step 4 | 缺 metrics / tracing / SLO | 先補 observability 再 deploy |
| Step 5 | Canary SLO 違反 | Auto-rollback + 開 incident |
