### 2026-05-08 - Identity material self-generation audit

Status: candidate

#### One-line Summary

Live-facing SDK/client work must classify required identity material by whether it can be self-generated, how it is generated, or which authorized provider owns it.

#### Human Explanation

Saying "authorized identity material remains" is not enough for development readiness. A live SDK or replay tool needs to know whether each device, install, account, session seed, vendor attestation, or server-issued session factor can be generated outside the target app runtime. If it can, the project docs need a sanitized generation recipe and validation matrix. If it cannot, the docs need a clear provider boundary such as caller-provided material, server-issued material, trusted bridge, private adapter, or unknown.

#### Trigger

Use this when API shape is known but live SDK/client/automation development still needs device, account, session, vendor, or server-issued runtime material.

#### Evidence

- Tool: Authorized APK analysis and SDK readiness documentation.
- Sanitized excerpt: A project had enough evidence to group identity-related keys, but the next development question required a stronger answer: whether each group can be generated by the SDK and, if yes, how.
- Evidence path: Keep concrete key names, hook windows, static traces, and live run results in project-local docs such as `<PROJECT_ROOT>/docs/domain-baseline.md`.

#### Generalized Lesson

For live development readiness, authorized identity material needs a self-generation audit, not only a sensitive-material inventory. Each key group must have a generation verdict, lifecycle/reset/cooldown behavior, and negative/error matrix status.

#### Agent Action

When identity material is needed for live access:

1. Create or update a project runtime baseline section named `Authorized identity material self-generation audit` or equivalent.
2. For each key group, record only names/shapes and classify it as `sdk-generatable`, `caller-provided`, `server-issued`, `trusted-bridge`, `private-adapter-required`, `unknown`, or `scoped-out`.
3. If `sdk-generatable`, document the sanitized generation recipe: inputs, algorithm family, canonical order, storage/refresh behavior, and fixture or parity validation.
4. If not SDK-generatable, document who owns it and what reset/cooldown/error behavior still blocks live development.
5. Do not allow live-facing SDK/client implementation to claim minimum readiness while required identity material is `unknown` or `private-adapter-required`, unless the scope is explicitly private adapter or trusted bridge.

#### Goal / Action / Validation

- Goal: Prevent live SDK/client development from starting with only API shape while required identity runtime factors remain undefined.
- Action: Promote identity material self-generation audit into the development readiness gate.
- Validation or reference source: Project docs must contain a table with verdict, generation recipe or provider boundary, lifecycle/reset/cooldown, error matrix, and sanitized evidence for each identity material group.

#### Applies When

- A project wants SDK, client, automation, replay, or live integration support.
- Runtime access depends on device/install/account/session/vendor/server-issued material.
- The analysis is trying to decide whether a feature is "ready except authorized identity material."

#### Does Not Apply When

- The task is offline parser, fixture, mock transport, or docs-only work that does not need live access.
- The material is explicitly scoped out of the current feature and cannot affect the requested live path.

#### Validation

Check that the project baseline can answer:

- Can this key group be generated without the target app runtime?
- If yes, what sanitized recipe and tests prove it?
- If no, who provides it and what lifecycle/reset/cooldown/error behavior is still open?
- Does the final readiness verdict distinguish private adapter readiness from standalone SDK self-generation?
