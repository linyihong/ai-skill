# SDK live self-generation audit

Status: candidate

## Trigger

A user asked whether an APK analysis had reached the stronger SDK readiness level where only caller-provided authorized identity material remains and every other live-fetch factor can be generated by the SDK itself. A concrete app's device id is only one example of that material, not the generic readiness name.

## Lesson

When an analysis is intended to support a live SDK, separate a **private adapter readiness gate** from a **SDK self-generation readiness gate**.

Private adapter readiness means the SDK core can call interfaces for base host, route binding, signing, session, opaque values, and decrypt, while raw material remains private or app-bound. SDK self-generation readiness is stricter: host selection, route/service binding, session/bootstrap, opaque fields, signing, decrypt, pagination, and error recovery must be deterministic from SDK code/config plus explicitly injected **authorized identity material**.

Authorized identity material means caller-provided or caller-initialized runtime identity such as a device id, install id, authorized test account, session seed, or legitimate device bootstrap result. If any runtime factor outside that category is still `private-adapter-required` or `unknown`, do not summarize the project as "only identity material remains." Summarize it as "SDK core plus private adapter required" and list the remaining non-identity blockers.

## Method

Add a runtime factor classification table to the project baseline or SDK-readiness document:

| Factor | Classification |
| --- | --- |
| base endpoint / host | `sdk-generatable` / `identity-material-bound` / `private-adapter-required` / `unknown` / `scoped-out` |
| route binding / service | same classification |
| authorized identity material | same classification |
| session/bootstrap | same classification |
| opaque query/header providers | same classification |
| signing/gateway | same classification |
| response decrypt/unwrap | same classification |
| error/session recovery | same classification |
| pagination/data truth | same classification |
| media/download, if in scope | same classification |

Then write a verdict:

```text
Live SDK self-generation verdict:
- ready except authorized identity material: yes/no
- remaining non-identity blockers: <factor list>
- allowed next work: live SDK implementation / private adapter only / offline parser only
```

## Evidence To Record

- Sanitized source or runtime evidence for each `sdk-generatable` factor.
- Authorized identity material type, lifecycle, reset behavior, and redaction rule.
- For private factors, only provider/interface names, key names, hashes/lengths, schema ids, and pass/fail results.
- Negative matrix status for bad device, stale session, bad signature, bad opaque value, and decrypt failure when authorized.

## Applicability

Use this when the user compares an APK target to another SDK that can already synthesize live requests, or when the next milestone is a standalone live SDK rather than offline parsers, mocks, or an app-bound private adapter.

## Safety

Do not put raw host lists, route service strings, signing keys, decrypted payload values, tokens, account data, or real device identifiers into reusable skill docs. Target-specific conclusions stay in project docs with sanitized classifications.
