# AI Codegen Passes CI / Fails Production（AI 生成程式碼通過 CI 但在 production 失敗）

Status: candidate
Class: `validation-gap` / `meta-tool-risk`

## Trigger

PR 大幅由 AI 生成（Copilot / Claude / Cursor / 自動化 agent）且觸動效能敏感路徑，但**未附 micro-benchmark、未經 reviewer perf checklist、未投資 production observability**。

具體觸發信號：

- Commit message / PR description 含 AI 標記（`Co-Authored-By: <model>`、"generated by"、"AI-assisted"）
- Diff 觸動熱路徑：API handler、DB 查詢層、外部 HTTP 呼叫、cache / session 累積結構
- PR 描述只提到「functional」，未提到 perf / load / latency / memory
- 同 PR 內無對應的 benchmark output、k6 / locust 結果、APM dashboard 連結
- Reviewer 心智頻寬已被 3.2x 產出稀釋，無法逐行人工檢查

## Failure Mode

AI 寫的程式碼結構上看起來合理，但對 load、邊界 case、長尾資料分布缺乏直覺。這導致：

1. **Unit test 高通過率 ≠ Production 安全**：43% AI 生成程式碼即使過 QA / staging 仍要在 production 手動 debug（參考 [`analysis/ai-augmented-delivery/ai-codegen-defect-distribution.md`](../../analysis/ai-augmented-delivery/ai-codegen-defect-distribution.md)）
2. **缺陷向下游集中**：unit / integration test 抓不到 perf 特性 → load 階段沒跑 → 上 production 才爆
3. **修復信任降低**：88% 公司需要 2–3 次 redeploy 才能確認 AI 修復有效
4. **Reviewer 過載**：3.2x 產出但 reviewer 心智頻寬不變 → 變相鼓勵「快速 approve」

## Risk

- Production incident 增加（latency 暴衝、OOM、connection pool 耗盡、SQL injection）
- 開發者 net 時間用在 debug 而非 build（38% 開發者每週約 2 天在 debug）
- 信任成本上升：對「測試通過」的訊號不再信任
- 工程組織把資源花在 hotfix 而非 feature

## Required Agent Action

當 PR 觸發以上信號時：

1. **靜態檢查 4 個 perf anti-pattern**（在 PR 階段，靜態 scan，cost 低）：
   - 迴圈內含 DB query / ORM 呼叫（N+1）
   - Long-lived map / slice 無 eviction（unbounded collection）
   - HTTP client / fetch 無 timeout 設定
   - SQL 字串拼接後接 keyword（injection + plan cache miss）

   詳見 [`validation/scenarios/software-delivery/ai-codegen-perf-risk-checklist.yaml`](../../validation/scenarios/software-delivery/ai-codegen-perf-risk-checklist.yaml)。

2. **熱路徑 PR-stage micro-benchmark**：對 API handler、DB layer、外部呼叫，要求附 benchmark output 或標註 `[skip-perf-bench]` 並寫理由。

3. **Reviewer perf risk checklist**：reviewer 不只看「功能對不對」，要明確列出「load 下會怎樣 / 邊界 case / 失敗 cascade」。

4. **Production observability gate**：APM、distributed tracing、SLO alerting 必須在 deploy 前先就位，否則該 PR 不應被 deploy 到 production。

## Prevention Gate

| 層級 | Gate | 機械化方式 |
|---|---|---|
| PR 靜態檢查 | 4 個 perf anti-pattern scan | Linter / AST scan / `validatePerfRisks`（候選 Phase 7） |
| PR human review | Reviewer perf checklist | PR template + `validation/scenarios/.../ai-codegen-perf-risk-checklist.yaml` |
| Pre-merge | Hot-path micro-benchmark | CI job（按熱路徑 path glob 觸發） |
| Pre-deploy | APM / SLO 就位 | Deploy pipeline gate |
| Post-deploy | Canary / progressive rollout | Feature flag + gradual % rollout |

對應 workflow：[`workflow/software-delivery/perf-risk-gate.md`](../../workflow/software-delivery/perf-risk-gate.md)。

Opt-out：`[skip-perf-bench]` 標註於 PR / commit body 並說明為什麼（純文件變更、UI 樣式調整、純測試新增等）。

## Validation

符合下列條件時，此 pattern 已被防止：

- PR 含 AI 標記且觸動熱路徑 → 4 個 perf anti-pattern 靜態檢查通過
- Reviewer 在 PR 上勾完 perf checklist（4 項）
- 熱路徑 PR 附 benchmark output 或 `[skip-perf-bench]` 說明
- 對應 production 路徑有 APM / SLO alert 設定
- 該 PR 部署後在 canary 階段觀察期內無 SLO 違反
