# 中性與低爭議文件用語

本規則適用於本庫所有 enforcement rules、skills、templates、feedback lessons，以及由 skill 產出的可重用文件。目的不是隱瞞事實，而是避免文件因用詞過度刺激、帶有判斷、像是在規避平台審查、或含有可被誤判的高風險詞而降低 AI 檢索、摘要、審查與協作穩定性。

## 核心原則

| 原則 | 做法 |
| --- | --- |
| 中性描述 | 用功能、角色、資料流、狀態、風險類別描述，不使用嘲諷、煽動、獵奇、誇大或道德判斷語。 |
| 授權語境 | 分析、測試、逆向、抓包、工具、SDK 等內容都要明確放在授權、合規、測試、文件化或相容性語境中。 |
| 降低觸發詞密度 | 高風險主題只在必要處以通用類別描述；不要在標題、檔名、slug、索引、摘要中反覆堆疊敏感詞。 |
| 保留可追溯性 | 若專案內必須保留原始 UI 文案、API 欄位、錯誤碼或法規術語，放在業務專案文件或證據附件；可重用 skill 文件使用泛化名稱與 placeholder。 |
| 不改變結論 | 中性化不能把未授權行為寫成授權行為，也不能把安全風險寫成已解決。若有風險，仍要明確描述控制、限制與驗證方法。 |
| 語言一致性 | 本庫可重用文件正文預設使用繁體中文；英文保留給路徑、指令、環境變數、API 欄位、程式符號與必要專有名詞。 |

## 寫作規則

- 標題、檔名、slug、索引列與 one-line summary 使用中性詞，例如 `media-flow`、`account-session`、`external-provider`、`risk-signal`、`content-module`、`feature-reconstruction`。
- 避免把特定品牌、產品、成人內容、仇恨、暴力、賭博、金融濫用、憑證、繞過、破解、攻擊、盜用等高風險詞放在可重用文件標題或檔名；必要時用「特定內容類別」「外部提供者」「授權測試」「風險操作」「敏感流程」「高風險端點」等中性分類。
- 原始證據若含高風險詞，只在專案私有文件中保留最小必要片段，並加上來源、授權範圍、去敏狀態與用途。不要把原始詞搬進 `enforcement/` 或 reusable skill lessons。
- 對外部服務、第三方 API、供應商、內容模組、帳號狀態、裝置狀態、付款或權限相關內容，優先描述「契約、欄位、狀態轉換、錯誤處理、驗證」而非刺激性場景。
- 避免使用「繞過、破解、盜刷、盜用、攻擊、入侵、洗錢、色情、暴力」這類詞作為文件標題、slug 或索引摘要；若必須討論，改寫成「未授權存取風險」「完整性風險」「交易濫用風險」「高風險內容類別」「授權安全測試」。
- 不要為了降低爭議而使用模糊到不可驗證的詞。每個中性詞仍要能對應到明確 evidence、contract、BDD、test 或 blocker question。
- 文件正文不要中英混雜成段落。除非文件本身是工具輸入範本、prompt、程式註解或外部規格引用，否則標題、摘要、說明段落與表格欄位優先使用繁體中文。
- 下列內容可保留英文：檔名、路徑、命令、環境變數、程式符號、API 欄位、固定 protocol 名稱、工具名稱、固定系統名稱，以及翻譯後會降低精準度的專有名詞，例如 `reference-first`、`Default Bootstrap`、`rule-weight`、`source-of-truth`。
- prompt template 可保留英文，方便直接貼給 agent；但 template 外的用途說明、前後文與驗證要求仍以繁體中文描述。

## 建議替換

| 容易造成誤判的寫法 | 優先寫法 |
| --- | --- |
| 目標 app 名稱、品牌名或內容站名 | `<target-app>`、`<content-provider>`、`target APK` |
| 成人、暴力、賭博等具體內容標籤 | 高風險內容類別、內容模組、媒體分類 |
| 破解、繞過、攻擊 | 授權分析、相容性測試、安全控制驗證、未授權存取風險 |
| 盜刷、洗錢、詐欺 | 交易濫用風險、風險交易、異常交易、風險訊號 |
| token / cookie 真值 | `<token-redacted>`、`<cookie-redacted>`、`<secret-redacted>` |
| 真實 host / endpoint | `<api-host>`、`<endpoint>`、`<provider-api>` |
| 真實帳號、裝置、個資 | `<account-id>`、`<device-id>`、`<user-data-redacted>` |

## 何時可以保留原詞

可以在專案私有文件或證據附件中保留最小必要原詞，但必須同時滿足：

1. 原詞是重現、合約、UI 對照、法規或錯誤碼判讀所必需。
2. 文件明確標示授權範圍與去敏狀態。
3. 不會被複製到 reusable skill、enforcement rule、lesson、模板或公開範例。
4. 可用中性別名在索引、摘要與 cross-link 中引用。

## Agent 輸出要求

產出或修改文件時：

1. 先判斷內容是否含高風險詞、私有產品名、真實 host、憑證、個資或容易被誤解為未授權行為的描述。
2. 在標題、檔名、slug、索引與摘要使用中性名稱。
3. 在正文用授權、合規、測試、相容性、契約或風險控制語境描述。
4. 將必要原始詞留在專案私有 evidence 區，並用 placeholder 或中性別名連回。
5. 若中性化可能改變技術含義，保留精準術語並加上授權、範圍與用途說明。
6. 檢查正文語言是否一致；若本檔主要面向本庫維護者，預設整理成繁體中文，僅保留必要英文術語與可直接複製的 prompt/code 區塊。

## 寫入前阻斷 Gate

修改本 repository 的可重用文件前，語言確認是 blocking gate，不是建議事項。

Agent 必須先完成並能回報：

1. **目標文件語言**：讀取目標文件標題、首段與相鄰章節，判斷正文主要語言。
2. **目標目錄政策**：確認目標路徑是否屬於 `enforcement/`、`workflow/`、`analysis/`、`intelligence/`、`feedback/`、`knowledge/`、`memory/`、`models/`、`governance/` 或其他可重用文件層。
3. **新內容語言**：新寫入的標題、摘要、說明段落、表格欄位與 checklist 必須跟隨目標文件語言；本庫可重用文件預設為繁體中文。
4. **保留原詞理由**：若保留英文、日文或其他語言片段，必須屬於路徑、命令、API 欄位、固定工具名、搜尋關鍵字、地名、法規/介面原文或翻譯後會降低精準度的專有名詞。
5. **寫入後檢查**：diff review 時必須檢查是否產生整句外語正文、混語段落或表格欄位語言不一致。

若上述任一項未完成，不得宣告文件更新完成，也不得進行 commit / push。若語言混雜是刻意保留，必須在 final response 或 commit message 中說明保留理由與範圍。
